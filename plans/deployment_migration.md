# План миграции и обхода блокировок для сервисов ИИ

## Анализ текущей архитектуры
На данный момент проект представляет собой SPA на Vite (React). Запросы к различным провайдерам ИИ (OpenRouter, AITunnel, ProxyAPI) идут не напрямую, а через серверную функцию (Vercel Serverless Function), которая находится в `api/proxy.js`.
Это правильный подход для скрытия API-ключей, однако при использовании зарубежных серверов могут возникать блокировки, задержки или ограничения по таймаутам (сейчас установлен лимит `maxDuration: 60` для Vercel).

## Варианты миграции и архитектурные решения

### 1. Cloudflare Pages + Cloudflare Pages Functions (Рекомендуемый)
Полный перенос и фронтенда, и прокси-апи на экосистему Cloudflare.
**Плюсы:**
* **Edge-вычисления:** Быстрая маршрутизация и исполнение кода по всему миру, что снижает задержку.
* **Обход блокировок:** Cloudflare обладает мощной и распределенной сетью. Использование их IP-адресов часто помогает обходить региональные ограничения к API LLM провайдеров (включая оригинальный API Google Gemini и Anthropic, если захотите отказаться от ProxyAPI).
* **Cloudflare AI Gateway:** Встроенный инструмент, который позволит анализировать затраты, кешировать типовые запросы (что сэкономит деньги) и автоматически делать fallback'и, если один из провайдеров упал (например, OpenRouter на AITunnel).
* **Таймауты:** В Cloudflare Workers/Pages Functions лимиты на время ожидания ответа от внешнего API мягче или могут быть лучше обработаны с помощью потоковой передачи (Streaming).

**Недостатки:**
* Нужно переписать `api/proxy.js` с формата Node.js (Req/Res) на формат Cloudflare Pages Functions (Web Standard `Request`/`Response` API).

### 2. Vercel + Cloudflare AI Gateway
Оставить проект на Vercel, но все запросы из `api/proxy.js` направлять не напрямую к провайдерам, а через шлюз Cloudflare AI Gateway.
**Плюсы:**
* Не нужно переписывать весь код прокси (только поменять URL-адреса в `fetch`).
* Получаем аналитику и кеширование.
**Недостатки:**
* Vercel (особенно Hobby-тариф) жестко ограничивает время работы Serverless Function 60 секундами (или 10 сек для Edge). Вызов LLM часто длится дольше (особенно Thinking models).
* Лишний хоп: Клиент -> Vercel -> Cloudflare -> Провайдер. Увеличивает задержку.

## Предлагаемый план действий (для режима Code)

1. **Создание директории `functions`:** 
   * Перенос логики из `api/proxy.js` в `functions/api/proxy.js` с использованием стандарта Cloudflare Pages Functions (`export async function onRequest(context) { ... }`).
2. **Настройка Cloudflare AI Gateway (опционально/рекомендуется):**
   * Модификация URL-адресов в `fetch` внутри прокси для прохождения через URL вашего AI Gateway (например: `https://gateway.ai.cloudflare.com/v1/{account_id}/{gateway_id}/openrouter/...`).
3. **Обновление скриптов деплоя:**
   * Добавление конфигурации для Wrangler (CLI-инструмент Cloudflare).
   * Создание файла `wrangler.toml` (если потребуется) или настройка переменных окружения через панель CF.
4. **Удаление старого кода:**
   * Удаление папки `api/` (предназначенной для Vercel), чтобы избежать конфликтов.
